<decisions>
## Implementation Decisions

### Edge Classification Rules
- Scope determined by **symbol project ownership**: if source symbol's project ≠ target symbol's project → `EdgeScope.CrossProject`
- Any different project = CrossProject, no distinction between direct and transitive dependencies
- Only create edges for **directly referenced types** in source code; do not walk inheritance hierarchies to create edges to ancestor projects
- Edge scope is **set at construction time** — every edge gets its scope when created, no post-classification pass

### External Stub Strategy
- **All non-solution types** are considered external (NuGet packages, .NET framework types, runtime types)
- Create **type-level stubs only** — one `SymbolNode` per external type, don't enumerate members
- **Filter out common primitives** (System.String, System.Int32, System.Threading.Tasks.Task, IEnumerable, etc.) — they add noise without documentation value
- Stub nodes carry **fully qualified name + containing assembly/package name** — enough to identify and link to docs later
- `NodeKind = NodeKind.Stub` and `IsExternal = true`

### SolutionSnapshot Structure
- SolutionSnapshot holds `ProjectEntry` records that **reference SymbolGraphSnapshot by ID** (not inline composition)
- Project-to-project dependencies modeled as **`ProjectEdge` records** with source/target project IDs and dependency kind — mirrors SymbolEdge pattern
- SolutionSnapshot is **immutable from creation** — use a builder pattern, consistent with existing SymbolGraphSnapshot design
- **Circular project references**: detect and warn (log), include both edges, don't crash — let downstream consumers handle cycles

### Test Coverage Approach
- **Dual approach**: synthetic unit tests with hand-built SymbolNodes/Edges for classification logic + integration tests with small real C# projects for end-to-end
- Priority edge cases: **multi-hop references** (A→B→C), **empty/no-reference projects**, and **diamond dependencies** (A→B,C; B,C→D)
- Stub node tests use **test-only dummy assemblies** — no coupling to specific NuGet package versions
- Existing 266+ tests validated through normal `dotnet test` run — no special regression gate
- Roslyn integration tests tagged with `[Trait("Category", "Integration")]` so they can be filtered

### Claude's Discretion
- Exact primitive/common type filter list
- Builder API design for SolutionSnapshot construction
- ProjectEdge property names and dependency kind enum values
- Warning format for circular reference detection
- Internal organization of classification logic (separate class vs inline)

</decisions>
