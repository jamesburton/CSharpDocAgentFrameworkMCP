After collecting edges, resolve ProjectOrigin for each edge's From and To nodes by looking them up in the snapshot. The approach:
   - After collecting edges into the `edges` list, get the current snapshot from `_query` (call `_query.GetSymbolAsync` for each unique From/To SymbolId, or more efficiently, access the snapshot nodes).
   - Since DocTools already has `_query` (IKnowledgeQueryService), and the snapshot is internal to KnowledgeQueryService, the simplest approach is: for each edge, call `_query.GetSymbolAsync(edge.From)` and `_query.GetSymbolAsync(edge.To)` to get the SymbolDetail which contains the node with ProjectOrigin. Cache these lookups to avoid redundant calls.
   - Build the JSON output as:
   ```csharp
   references = edges.Select(e => new {
       fromId = e.From.Value,
       toId = e.To.Value,
       edgeKind = e.Kind.ToString(),
       scope = e.Scope.ToString(),
       fromProject = nodeProjectCache.GetValueOrDefault(e.From),
       toProject = nodeProjectCache.GetValueOrDefault(e.To)
   }).ToList()
   ```
   Where `nodeProjectCache` is a `Dictionary<SymbolId, string?>` built by resolving unique From/To ids via `_query.GetSymbolAsync`.

3. **GetSymbol method — FQN disambiguation** (TOOLS-02):
   The locked decision states: "Ambiguous fully qualified names (same FQN in multiple projects) require project qualifier; return error listing which projects have the symbol."

   Currently `get_symbol` takes a `symbolId` parameter. Add FQN resolution:
   - After the existing null check for `symbolId`, before calling `_query.GetSymbolAsync`, add FQN detection logic:
   - If the `symbolId` does NOT look like a stable SymbolId (heuristic: stable SymbolIds contain `|` separator from assembly-qualified format), treat it as a potential FQN.
   - For FQN resolution: use `_query.SearchAsync(symbolId, limit: 50)` to find candidate symbols, then filter results where the underlying node's `FullyQualifiedName` matches the input exactly. Since SearchResultItem doesn't expose FQN, instead scan snapshot nodes. The practical approach:
     - Add a new private helper method `ResolveByFqnAsync(string fqn, CancellationToken ct)` that calls `_query.SearchAsync(fqn, limit: 100)` and then for each result calls `_query.GetSymbolAsync(result.Id)` to check if `FullyQualifiedName == fqn`.
     - If exactly 1 match: proceed with that SymbolId.
     - If 0 matches: fall through to existing GetSymbolAsync (which will return NotFound).
     - If 2+ matches: group by `ProjectOrigin` and return an error response: `"Ambiguous FQN '{fqn}' found in multiple projects: {projectList}. Specify the stable SymbolId or use search_symbols with project filter to disambiguate."`
   - This preserves backward compat: existing SymbolId-based calls (containing `|`) skip FQN resolution entirely.

4. **GetSymbolFqnDisambiguationTests.cs** — Create test file with:

   - `GetSymbol_WithUniqueFqn_ResolvesSuccessfully` — FQN that exists in only one project resolves to that symbol.
   - `GetSymbol_WithAmbiguousFqn_ReturnsErrorListingProjects` — FQN existing in ProjectA and ProjectB returns error mentioning both project names.
   - `GetSymbol_WithUnknownFqn_ReturnsNotFound` — FQN that doesn't exist returns NotFound.
