Add a new section `## TypeScript Sidecar Architecture` to docs/Architecture.md, placed after the existing project-level sections (after the "Ingestion Pipeline" or similar section, before any appendix). The section must cover three topics required by the verification:

**1. Node.js Sidecar Architecture:**
- `ts-symbol-extractor` is a standalone Node.js ESM project in `src/ts-symbol-extractor/`
- Bundled to `dist/index.js` via esbuild for single-file deployment
- C# `TypeScriptIngestionService` spawns the sidecar as a child process
- Communication via stdin/stdout; all sidecar logging goes to stderr
- Sidecar uses TypeScript Compiler API (`ts.createProgram`) to parse projects

**2. NDJSON Protocol Definition:**
- Request: JSON-RPC 2.0 over stdin, one JSON object per line
- Method: `"extract"` with params `{ tsconfigPath: string, outputPath?: string }`
- Response: JSON-RPC 2.0 response with `result` containing a `SymbolGraphSnapshot`-shaped object
- For large responses, `outputPath` triggers file-based output to bypass pipe buffer limits
- Error responses use standard JSON-RPC error codes (-32700 parse, -32600 invalid, -32601 method not found, -32603 internal)

**3. TypeScript Symbol Mapping Strategy:**
- TypeScript source files map to `SymbolKind.Namespace` nodes using relative file paths
- SymbolId format: `{prefix}:{projectName}:{relativePath}:{symbolName}` (e.g., `T:myapp:src/utils.ts:Helper`)
- Prefixes: N (namespace/file), T (type/class/interface/enum/type alias), M (method/function), P (property), F (field), E (event/enum member)
- Inheritance: `extends` maps to `InheritsFrom` edges, `implements` maps to `Implements` edges
- JSDoc/TSDoc maps to `DocComment` fields (summary, param, returns, example, throws, see, remarks)
- Export visibility: exported = Public, non-exported = Internal
- `node_modules` and `.d.ts` files are excluded from extraction

Keep the section concise (40-80 lines of markdown). Use tables where appropriate. Reference actual file paths.
