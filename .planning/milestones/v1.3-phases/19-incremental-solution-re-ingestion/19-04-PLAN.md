  <action>
Replace lines 103-126 of IncrementalSolutionIngestionService.cs (the block that says "For now, delegate to full ingestion") with the actual incremental skip logic. The production path MUST call:

1. `DependencyCascade.HasStructuralChange(previousSnapshot.Projects, currentProjectPaths)` — if true, fall through to full ingest (existing behavior, but now with explicit structural check).

2. For each project in `previousSnapshot.Projects`:
   - Call `SolutionManifestStore.LoadAsync(artifactsDir, slnPath, project.Path, ct)` to get previous manifest
   - Call `SolutionManifestStore.ComputeProjectManifestAsync(project.Path, refPaths, chosenTfm, ct)` to get current manifest
   - Call `FileHasher.Diff(previousManifest, currentManifest)` to check if changed
   - Collect directly-changed project names

3. Build `ProjectEdge` list from `previousSnapshot.ProjectDependencies`.

4. Call `DependencyCascade.ComputeDirtySet(directlyChanged, projectEdges)` to get full dirty set.

5. Call `DependencyCascade.TopologicalSort(previousSnapshot.Projects, projectEdges)` for processing order.

6. Iterate projects in topological order:
   - If NOT in dirty set: emit `ProjectIngestionStatus(name, path, "skipped", "unchanged", nodeCount, tfm)` — preserve that project's nodes/edges from `previousSnapshot`.
   - If in dirty set: delegate to `_fullIngestionService.IngestAsync` for full re-ingest (this is the simplest correct approach — a dirty project triggers full solution re-ingest via the existing service, since per-project re-compilation requires MSBuildWorkspace which only the full service manages).

**CRITICAL SIMPLIFICATION:** Since `SolutionIngestionService` owns the MSBuildWorkspace and per-project compilation pipeline, and MSBuildWorkspace is not thread-safe, the cleanest production path for dirty projects is:
- If dirty set is EMPTY (nothing changed) → build result from previous snapshot with all projects marked "skipped"
- If dirty set is NON-EMPTY → delegate to `_fullIngestionService.IngestAsync` (same as structural change)

This means the production skip path's VALUE is avoiding full re-ingest when NOTHING changed (the common case for `ingest_solution` re-calls). The dependency cascade still matters for the result metadata (which projects WOULD need re-ingest) and for future optimization.

**For the "nothing changed" path:**
- Create `SolutionIngestionResult` from `previousSnapshot`:
  - `SnapshotId` = previous snapshot's content hash (from pointer file)
  - `SolutionName` = from previous snapshot
  - `TotalProjectCount` = previous snapshot project count
  - `IngestedProjectCount` = 0 (all skipped)
  - `TotalNodeCount` / `TotalEdgeCount` from previous snapshot
  - `Projects` = list of `ProjectIngestionStatus` with Status="skipped", Reason="unchanged" for each
  - `Snapshot` = previousSnapshot (reuse as-is)
- Do NOT re-save the snapshot (it already exists)
- Do NOT re-save manifests (they haven't changed)
- Log: "All projects unchanged for {Solution} — returning cached snapshot"

**For the "something changed" path:**
- Log: "Detected {N} dirty projects for {Solution} — performing full re-ingest"
- Delegate to `_fullIngestionService.IngestAsync` (existing fallback)
- Save pointer + manifests as before

**Stub lifecycle:** When returning the cached snapshot (nothing changed), stubs are preserved automatically because the entire previous snapshot is reused. When delegating to full ingest, stubs are regenerated from scratch by the full service (existing correct behavior).

Keep the existing `PipelineOverride` seam, `forceFullReingest` path, `LoadPreviousSnapshotAsync`, `SavePointerAsync`, `SaveProjectManifestsAsync`, and `EmitTelemetry` methods unchanged.
