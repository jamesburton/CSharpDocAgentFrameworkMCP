## User Constraints (from CONTEXT.md)

### Locked Decisions

#### Edge Classification Rules
- Scope determined by **symbol project ownership**: if source symbol's project ≠ target symbol's project → `EdgeScope.CrossProject`
- Any different project = CrossProject, no distinction between direct and transitive dependencies
- Only create edges for **directly referenced types** in source code; do not walk inheritance hierarchies to create edges to ancestor projects
- Edge scope is **set at construction time** — every edge gets its scope when created, no post-classification pass

#### External Stub Strategy
- **All non-solution types** are considered external (NuGet packages, .NET framework types, runtime types)
- Create **type-level stubs only** — one `SymbolNode` per external type, don't enumerate members
- **Filter out common primitives** (System.String, System.Int32, System.Threading.Tasks.Task, IEnumerable, etc.) — they add noise without documentation value
- Stub nodes carry **fully qualified name + containing assembly/package name** — enough to identify and link to docs later
- `NodeKind = NodeKind.Stub` and `IsExternal = true`

#### SolutionSnapshot Structure
- SolutionSnapshot holds `ProjectEntry` records that **reference SymbolGraphSnapshot by ID** (not inline composition)
- Project-to-project dependencies modeled as **`ProjectEdge` records** with source/target project IDs and dependency kind — mirrors SymbolEdge pattern
- SolutionSnapshot is **immutable from creation** — use a builder pattern, consistent with existing SymbolGraphSnapshot design
- **Circular project references**: detect and warn (log), include both edges, don't crash — let downstream consumers handle cycles

#### Test Coverage Approach
- **Dual approach**: synthetic unit tests with hand-built SymbolNodes/Edges for classification logic + integration tests with small real C# projects for end-to-end
- Priority edge cases: **multi-hop references** (A→B→C), **empty/no-reference projects**, and **diamond dependencies** (A→B,C; B,C→D)
- Stub node tests use **test-only dummy assemblies** — no coupling to specific NuGet package versions
- Existing 266+ tests validated through normal `dotnet test` run — no special regression gate
- Roslyn integration tests tagged with `[Trait("Category", "Integration")]` so they can be filtered

### Claude's Discretion
- Exact primitive/common type filter list
- Builder API design for SolutionSnapshot construction
- ProjectEdge property names and dependency kind enum values
- Warning format for circular reference detection
- Internal organization of classification logic (separate class vs inline)

None — discussion stayed within phase scope.

## Summary

Phase 14.1 is a pure **enrichment pass** on top of the existing `SolutionIngestionService`. All domain types (`SolutionSnapshot`, `ProjectEntry`, `ProjectEdge`, `SymbolEdge`, `NodeKind`, `EdgeScope`) were defined in Phase 13 and exist today in `DocAgent.Core`. The current `SolutionIngestionService.IngestAsync` already opens the Roslyn `Solution`, processes each `Project`, and accumulates nodes/edges — but it produces a flat `SymbolGraphSnapshot` without populating `SolutionSnapshot`, without classifying edge scopes, and without synthesizing stub nodes for external types.

The work is three additive capabilities layered on the existing walk loop: (1) populate a `SolutionSnapshot` builder alongside the existing flat snapshot, (2) determine `EdgeScope` at edge-construction time using `ProjectOrigin` lookup, and (3) during `WalkTypeInline`, detect external type references (`ContainingAssembly` not in solution projects), deduplicate them with a `HashSet<string>`, and emit `NodeKind.Stub` nodes. The `PipelineOverride` seam and the existing `WalkNamespaceInline`/`WalkTypeInline` pattern in `SolutionIngestionService` give natural extension points without touching `RoslynSymbolGraphBuilder`.

**Primary recommendation:** Extend `SolutionIngestionService.IngestAsync` (and `WalkTypeInline`) in three focused changes — snapshot builder population, scope classification, and stub synthesis — keeping everything inside the service class consistent with the inline-walk decision from Phase 14-01.

### Pattern 1: SolutionSnapshot Builder Population

**What:** After processing all projects, build `SolutionSnapshot` from accumulated per-project data and persist via `SnapshotStore` (if a `SaveSolutionAsync` method exists) or store inline.

**Current state:** `SolutionIngestionService.IngestAsync` only builds a flat `SymbolGraphSnapshot`. The `SolutionSnapshot` type exists in Core but is never instantiated by the ingestion service.

**How to build:** During the project loop, accumulate:
- `List<ProjectEntry>` — built from `project.FilePath`, `project.Name`, and `project.ProjectReferences`
- `List<ProjectEdge>` — from `project.ProjectReferences` → look up target project name
- `List<SymbolGraphSnapshot>` — per-project snapshots (create per-project `SymbolGraphSnapshot` with that project's nodes/edges)

After the loop, construct `SolutionSnapshot` and persist it.

**Key Roslyn API:**
```csharp
// Get direct project-to-project dependencies
foreach (var projRef in project.ProjectReferences)
{
    var target = solution.GetProject(projRef.ProjectId);
    if (target is not null)
        projectEdges.Add(new ProjectEdge(projectName, targetName));
}
```

**Confidence:** HIGH — `Project.ProjectReferences` is a standard Roslyn API, `Solution.GetProject(ProjectId)` resolves it.

### Pattern 2: EdgeScope Classification at Construction Time

**What:** When creating a `SymbolEdge` during `WalkTypeInline`, determine if the target symbol belongs to the current project or a different solution project.

**Current state:** `WalkTypeInline` creates all edges with the default `EdgeScope.IntraProject`. The target node for cross-project/external edges may not exist in the current project's nodes list.

**How to classify:**
```csharp
// Build lookup during project loop — available before WalkTypeInline is called
var solutionProjectNames = new HashSet<string>(
    chosenProjects.Select(p => GetProjectName(p)), StringComparer.OrdinalIgnoreCase);

// In WalkTypeInline, when creating Inherits/Implements edge:
private static EdgeScope ClassifyScope(ITypeSymbol targetType, string currentProjectName, HashSet<string> solutionProjectNames)
{
    // External: containing assembly not in solution
    if (targetType.ContainingAssembly is null) return EdgeScope.External;
    var targetAssembly = targetType.ContainingAssembly.Name;
    if (!solutionProjectNames.Contains(targetAssembly)) return EdgeScope.External;
    // Cross-project: same solution but different project
    if (!string.Equals(targetAssembly, currentProjectName, StringComparison.OrdinalIgnoreCase))
        return EdgeScope.CrossProject;
    return EdgeScope.IntraProject;
}
```

**Note:** The `WalkTypeInline` private static method currently takes no project context. It needs `currentProjectName` and `solutionProjectNames` threaded through, or captured via a local helper class/closure.

**Confidence:** HIGH — `ITypeSymbol.ContainingAssembly` is a core Roslyn API. Assembly name maps to project name in the solution.

### Pattern 3: External Stub Node Synthesis

**What:** During `WalkTypeInline`, when processing `BaseType` or `Interfaces`, if the referenced type is external, create a `SymbolNode` with `NodeKind = NodeKind.Stub`.

**Current state:** `WalkTypeInline` creates edges to external types by ID but never creates corresponding nodes. This leaves dangling edge endpoints.

**How to synthesize:**
```csharp
// Thread a shared stub accumulator through the walk
private static void MaybeAddStubNode(
    ITypeSymbol externalType,
    List<SymbolNode> stubNodes,
    HashSet<string> seenStubIds)
{
    if (IsPrimitive(externalType)) return; // filter noise
    var id = GetSymbolId(externalType);
    if (!seenStubIds.Add(id.Value)) return; // deduplicate
    stubNodes.Add(new SymbolNode(
        Id: id,
        Kind: SymbolKind.Type,
        DisplayName: externalType.Name,
        FullyQualifiedName: externalType.ToDisplayString(),
        PreviousIds: Array.Empty<SymbolId>(),
        Accessibility: Accessibility.Public,
        Docs: null,
        Span: null,
        ReturnType: null,
        Parameters: Array.Empty<ParameterInfo>(),
        GenericConstraints: Array.Empty<GenericConstraint>(),
        ProjectOrigin: externalType.ContainingAssembly?.Name,
        NodeKind: NodeKind.Stub));
}
```

**Primitive filter (Claude's discretion — recommended list):**
```csharp
private static readonly HashSet<string> PrimitiveTypeNames = new(StringComparer.Ordinal)
{
    "System.Object", "System.String", "System.Boolean",
    "System.Byte", "System.SByte", "System.Int16", "System.UInt16",
    "System.Int32", "System.UInt32", "System.Int64", "System.UInt64",
    "System.Single", "System.Double", "System.Decimal",
    "System.Char", "System.Void", "System.IntPtr", "System.UIntPtr",
    "System.Threading.Tasks.Task", "System.Threading.Tasks.Task`1",
    "System.Collections.Generic.IEnumerable`1",
    "System.Collections.Generic.IList`1",
    "System.Collections.Generic.IReadOnlyList`1",
    "System.Collections.Generic.ICollection`1",
    "System.Collections.Generic.IDictionary`2"
};

private static bool IsPrimitive(ITypeSymbol type)
    => PrimitiveTypeNames.Contains(type.ToDisplayString());
```

**Confidence:** HIGH — approach consistent with `CreateSimpleNode` pattern already in the service.

### Pattern 4: GRAPH-05 — Stub Node Filtering in Index

**What:** `BM25SearchIndex.IndexAsync` and `InMemorySearchIndex.IndexAsync` must skip nodes with `NodeKind == NodeKind.Stub` to prevent search pollution.

**Current state:** Both index implementations call `IndexAsync(SymbolGraphSnapshot snapshot, ...)` and process all nodes. The `NodeKind` discriminator exists on `SymbolNode` but is not checked.

**How to add:**
```csharp
// In IndexAsync, before processing nodes:
var nodesToIndex = snapshot.Nodes.Where(n => n.NodeKind == NodeKind.Real).ToList();
```

**Confidence:** HIGH — `SymbolNode.NodeKind` is already the defaulted field in Core; adding the filter is a one-liner.

### Anti-Patterns to Avoid

- **Post-classification pass:** Do not add a second loop over edges to set scope after all walks complete. Locked decision: scope set at construction time.
- **Walking inheritance ancestors for cross-project edges:** Only directly referenced types (base class, interfaces) create edges. Do not recurse to grandparent base classes.
- **Storing `SymbolGraphSnapshot` per project and then merging:** The existing flat-merge approach (`allNodes`, `allEdges`) is deliberate. Per-project snapshots go into `SolutionSnapshot.ProjectSnapshots`; the flat snapshot for MCP tool consumption stays as-is.
- **Opening a second `MSBuildWorkspace` per project:** Phase 14-01 decision — solution ingestion uses the solution-scoped workspace only. `WalkNamespaceInline` was introduced specifically to avoid this.

## Common Pitfalls

### Pitfall 1: `WalkTypeInline` is a private static — cannot close over instance state

**What goes wrong:** The classification helpers (`ClassifyScope`, `MaybeAddStubNode`) need per-ingestion context (current project name, solution project set, stub accumulator). Private static methods cannot capture these without parameter threading.

**Why it happens:** `WalkTypeInline` and `WalkNamespaceInline` are static to avoid accidental state sharing between tests. Adding parameters risks signature explosion.

**How to avoid:** Introduce a small `ProjectWalkContext` struct or record (internal, non-public) holding `CurrentProjectName`, `SolutionProjectNames`, `StubAccumulator`, and thread it through the walk methods:
```csharp
private readonly record struct ProjectWalkContext(
    string ProjectName,
    HashSet<string> SolutionProjectNames,
    List<SymbolNode> StubNodes,
    HashSet<string> SeenStubIds);
```
Pass by value (or `ref` for the mutable lists). All walk methods become `static ... (... , in ProjectWalkContext ctx)`.

### Pitfall 2: Generic type arguments create duplicate stubs

**What goes wrong:** `IEnumerable<MyType>` produces a separate `INamedTypeSymbol` for the constructed generic. `INamedTypeSymbol.OriginalDefinition` is the unbound generic; naively adding both creates duplicates.

**How to avoid:** When synthesizing stubs, always use `type.OriginalDefinition` for the ID check, or use `type.ToDisplayString()` consistently. The `HashSet<string> seenStubIds` deduplication handles this if ID generation is consistent.

### Pitfall 3: `Project.ProjectReferences` uses `ProjectId`, not name

**What goes wrong:** `project.ProjectReferences` yields `ProjectReference` objects with a `ProjectId` GUID. Looking up the target project name requires `solution.GetProject(projRef.ProjectId)?.Name`.

**How to avoid:** Build a `Dictionary<ProjectId, string>` from `chosenProjects` before the dependency-edge loop:
```csharp
var projectIdToName = chosenProjects.ToDictionary(
    p => p.Id, p => GetProjectName(p));
```

### Pitfall 4: Multi-targeted project deduplication means fewer Roslyn `Project` objects

**What goes wrong:** After TFM deduplication, `chosenProjects` contains one `Project` per `.csproj`. But `project.ProjectReferences` may reference a `ProjectId` that was de-selected (a sibling TFM). Resolving that ID via `solution.GetProject()` returns the Roslyn project object, which may have a TFM-suffixed name.

**How to avoid:** Use `project.FilePath` as the stable key for project identity rather than `project.Name` (which may include a TFM suffix). Build `projectIdToName` using the deduplicated file-path → clean name mapping already constructed in the existing `tfmChoices` dictionary.

### Pitfall 5: Circular reference detection needs visited tracking per DFS

**What goes wrong:** Simply checking `if (From == To)` is insufficient for a cycle. Projects A→B→A form a cycle not detected by self-loop check.

**How to avoid:** After building all `ProjectEdge`s, run a simple DFS/topological sort for cycle detection. Log a warning per detected cycle. Do not crash — include both edges as per locked decision.

```csharp
private static void WalkTypeInline(
    INamedTypeSymbol type,
    SymbolId typeId,
    List<SymbolNode> nodes,
    List<SymbolEdge> edges)
{
    // Inheritance edge
    if (type.BaseType is not null && type.BaseType.SpecialType != SpecialType.System_Object)

    {
        edges.Add(new SymbolEdge(typeId, GetSymbolId(iface), SymbolEdgeKind.Implements));
        // ← PHASE 14.1: classify scope, synthesize stub if external
    }
    // ...
}
```

### Project dependency extraction (new)
```csharp
// During chosenProjects loop in IngestAsync
var projectIdToName = chosenProjects
    .Where(p => !string.IsNullOrEmpty(p.FilePath))
    .ToDictionary(p => p.Id, p => Path.GetFileNameWithoutExtension(p.FilePath!));

var projectEdges = new List<ProjectEdge>();
var projectEntries = new List<ProjectEntry>();

foreach (var project in chosenProjects)
{
    var projName = GetProjectName(project);
    var directDeps = project.ProjectReferences
        .Where(r => projectIdToName.ContainsKey(r.ProjectId))
        .Select(r => projectIdToName[r.ProjectId])
        .ToList();

    projectEntries.Add(new ProjectEntry(projName, project.FilePath ?? string.Empty, directDeps));
    foreach (var dep in directDeps)
        projectEdges.Add(new ProjectEdge(projName, dep));
}
```

## Open Questions

1. **Does `SnapshotStore` need a `SaveSolutionAsync` overload for `SolutionSnapshot`?**
   - What we know: `SnapshotStore` currently has `SaveAsync(SymbolGraphSnapshot)`. `SolutionSnapshot` is a distinct type.
   - What's unclear: Whether Phase 14.1 should persist `SolutionSnapshot` via MessagePack to disk, or treat it as an in-memory aggregate returned from `IngestAsync` for MCP tool consumption.
   - Recommendation: Add `SaveSolutionSnapshotAsync(SolutionSnapshot)` to `SnapshotStore` consistent with the existing pattern. If out of scope for this phase, at minimum return `SolutionSnapshot` from `IngestAsync` (requires changing `SolutionIngestionResult` or `ISolutionIngestionService`).

2. **`ISolutionIngestionService.IngestAsync` returns `SolutionIngestionResult`, not `SolutionSnapshot`**
   - What we know: The current return type does not expose a `SolutionSnapshot`.
   - What's unclear: Whether Phase 14.1 requires adding `SolutionSnapshot?` to `SolutionIngestionResult`, or if the snapshot is only persisted.
   - Recommendation: Add `SolutionSnapshot? Snapshot` to `SolutionIngestionResult` (nullable for backward compat) and persist via `SnapshotStore`. This avoids changing the interface contract materially.

3. **`IsExternal` property on `SymbolNode`**
   - What we know: The CONTEXT.md says stub nodes carry `IsExternal = true`, but `SymbolNode` in `Symbols.cs` has no `IsExternal` property — only `NodeKind = NodeKind.Stub`.
   - What's unclear: Whether `IsExternal` was meant to be `NodeKind == NodeKind.Stub` as a computed property, or a separate `bool IsExternal` field was planned but not yet added.
   - Recommendation: `NodeKind.Stub` already encodes externality. Do not add a redundant `bool IsExternal` field. Use `NodeKind` as the sole discriminator, consistent with Phase 13 design and GRAPH-05 filter.
