# Pitfalls Research

**Domain:** NuGet Package Mapping — lock file parsing, DLL reflection via MetadataReference, PackageGraph structure, stub node enrichment
**Researched:** 2026-03-26
**Confidence:** HIGH (lock file format from NuGet wiki + official docs; MetadataReference memory behavior from Roslyn docs and tracked issues; cache paths from official Microsoft Learn; TFM selection from NuGet docs; stub/FQN analysis from direct codebase inspection)

---

## Critical Pitfalls

### Pitfall 1: packages.lock.json Is Opt-In — Most Projects Won't Have One

**What goes wrong:**
The parser is built to read `packages.lock.json` and the milestone ships, but every target repo that is not explicitly opted in will fail silently or throw file-not-found. The tool returns empty dependency data without explaining why. `get_dependencies` shows zero packages for a project that clearly has 40 PackageReferences.

**Why it happens:**
`RestorePackagesWithLockFile=true` must be set in the project file (or a lock file must already exist at the project root). The file is not generated by default on `dotnet restore`. Developers assume it works like npm's `package-lock.json`, which is always present. This repo uses `ManagePackageVersionsCentrally=true` with `Directory.Packages.props` and does not set `RestorePackagesWithLockFile`, so it has no `packages.lock.json` files.

**How to avoid:**
Use `project.assets.json` (in `obj/`) as the primary dependency source. It is always present after any restore, contains the same fully resolved transitive graph including versions, and is the canonical file MSBuild uses. Fall back to `packages.lock.json` only when it exists alongside the `.csproj` and `project.assets.json` is absent. Document that `get_dependencies` reads `project.assets.json`, not `packages.lock.json`. Version 2 of packages.lock.json is generated when Central Package Management is enabled — relevant for this repo, but not reliable across all consumers.

If `packages.lock.json` is the only source used, the `ingest_project` flow must emit a diagnostic when neither file is present rather than returning an empty graph.

**Warning signs:**
- `get_dependencies` returns zero packages for projects that clearly have PackageReferences
- Integration tests pass because the test fixture includes a pre-committed lock file but the actual DocAgentFramework projects do not

**Phase to address:** Lock file / dependency source selection — first decision before any parsing logic is written.

### Pitfall 2: packages.lock.json TFM Key Is the Canonical Long Form, Not the Short TFM

**What goes wrong:**
Code parses the `dependencies` object using the short TFM (`net10.0`) as the key and finds nothing. The actual key written by `dotnet restore` is the canonical long form: `.NETCoreApp,Version=v10.0`. For a multi-targeting project with a RID, the key is `.NETCoreApp,Version=v10.0/linux-x64`. These strings never match with simple equality.

**Why it happens:**
The NuGet wiki spec example shows a short form (`netcore2.0`) that differs from what the NuGet client actually writes. Real lock files generated by `dotnet restore` use the canonical long form. The mismatch is invisible until you compare a hand-crafted test fixture against a real lock file.

Version 2 lock files (generated when CPM is active, which this repo uses) maintain the same long-form TFM key schema.

**How to avoid:**
Do not match TFM keys by exact string equality against the short form. Use `NuGetFramework.Parse("net10.0").DotNetFrameworkName` to derive the canonical key (`.NETCoreApp,Version=v10.0`) for lookup. Alternatively, parse each key using `NuGetFramework.ParseFolder()` and compare using `NuGetFramework.Equals()`. If `NuGet.Frameworks` is not a project dependency, write a minimal local normalizer — the mapping for `netN.M` to `.NETCoreApp,Version=vN.M` is a small static table.

**Warning signs:**
- Parser returns empty dependency map despite lock file existing
- Unit test with a hand-crafted fixture passes but a real `packages.lock.json` from this repo fails
- The key appears in the JSON file but Dictionary lookup returns null

**Phase to address:** Lock file parsing — write the TFM normalizer before writing any dependency extraction logic.

### Pitfall 3: NuGet Global Cache Path Has Four Override Layers — Hard-Coding the Default Breaks on CI

**What goes wrong:**
Code hard-codes `Path.Combine(Environment.GetFolderPath(Environment.SpecialFolder.UserProfile), ".nuget", "packages")` as the NuGet cache root. This path does not exist on CI agents with `NUGET_PACKAGES` set to `/root/.nuget/packages` or `D:\a\.nuget\packages`. It also breaks on machines with a solution-level `nuget.config` that sets `globalPackagesFolder` to a shared network location.

**Why it happens:**
The default path is well-known and works on developer machines. The four-level override chain is invisible from code that ignores it: `NUGET_PACKAGES` env var → nuget.config `globalPackagesFolder` → `RestorePackagesPath` MSBuild property → OS-specific default. CI environments almost always override the default.

Official paths per platform (fallback only): Windows `%userprofile%\.nuget\packages`, Mac/Linux `~/.nuget/packages`.

**How to avoid:**
Use `NuGet.Configuration.SettingsUtility.GetGlobalPackagesFolder(settings)` with `Settings.LoadDefaultSettings(solutionDir)`. This resolves the full override chain including nuget.config at the solution root. The `NuGet.Configuration` package is already a transitive dependency of `Microsoft.CodeAnalysis.Workspaces.MSBuild` so no new dependency is needed.

As a lighter alternative: check `NUGET_PACKAGES` env var first, then fall through to the platform default.

**Warning signs:**
- DLL reflection throws `FileNotFoundException` on CI even though packages restored successfully
- Works on developer machine, fails in Docker build container
- Works on Windows CI, fails on Linux CI

**Phase to address:** DLL path resolution — establish the cache root resolver before any path construction code.

### Pitfall 4: DLL-in-Cache Path Construction Requires a TFM Best-Fit Walk, Not Direct TFM Substitution

**What goes wrong:**
Code constructs `{globalPackages}/{packageId}/{version}/lib/{projectTfm}/{assembly}.dll` substituting the project's TFM directly. The file does not exist because the package ships `lib/netstandard2.0/` or `lib/net8.0/` but has no `lib/net10.0/` subfolder. `MetadataReference.CreateFromFile` throws `FileNotFoundException` for every package that predates .NET 10.

**Why it happens:**
NuGet applies a best-fit compatibility algorithm at restore time to select the DLL from the closest compatible TFM folder. The code must replicate this: enumerate the `lib/` subfolders, parse each as a `NuGetFramework`, filter to those compatible with the project's TFM, then pick the highest compatible version. Some packages split compile-time reference assemblies into `ref/{tfm}/` — NuGet prefers `ref/` over `lib/` for compilation, and `ref/` assemblies are correct for API surface extraction.

Example: `YamlDotNet 16.3.0` ships only `lib/netstandard2.0/` — there is no `lib/net10.0/`. The best-fit result is `lib/netstandard2.0/YamlDotNet.dll`.

**How to avoid:**
Enumerate `ref/{tfm}/` first, then `lib/{tfm}/`. For each, list subdirectories, parse each name as `NuGetFramework.ParseFolder()`, filter using `DefaultCompatibilityProvider.Instance.IsCompatible(projectTfm, candidateTfm)`, sort descending, take the first. Load the DLL from that subfolder. If `NuGet.Frameworks` is available (it is as a transitive dep), use its APIs directly. If not, a lightweight compatibility table for the common TFM hierarchy is sufficient.

**Warning signs:**
- `MetadataReference.CreateFromFile` throws for packages that clearly have DLLs in the cache
- Works for packages that ship `net10.0` explicitly (e.g., `Microsoft.Extensions.Hosting 10.0.0`) but fails for `netstandard2.0`-only packages
- Works in tests that use a mock path but fails against the real NuGet cache

**Phase to address:** DLL resolution — implement and unit-test the TFM best-fit selector against packages from this project's own cache before wiring into the ingestion pipeline.

### Pitfall 5: Loading Many DLLs via MetadataReference.CreateFromFile Leaks Native Memory

**What goes wrong:**
`MetadataReference.CreateFromFile(path)` is called per assembly per ingestion call without disposal. A solution with 30 projects, each referencing 50 packages, produces 150+ unique DLL paths. Each call reads the full PE image into native (non-GC) heap. In a long-running MCP server, this heap grows monotonically because the GC has no visibility into native memory pressure and does not reclaim it promptly. Documented cases report 250MB leaks per compilation cycle.

**Why it happens:**
The single-argument `MetadataReference.CreateFromFile(string path)` overload does not return an `IDisposable`, so callers have no explicit cleanup path. The native block is released only when the GC finalizes the internal object, which can be delayed indefinitely in a server process with low allocation pressure.

**How to avoid:**
Use the two-step `AssemblyMetadata` pattern for bulk loading:

```csharp
using var metadata = AssemblyMetadata.CreateFromFile(dllPath);
var reference = metadata.GetReference();
// use reference synchronously, then metadata.Dispose() is called on exit
```

For the MCP server's ingestion pipeline, implement a `MetadataReferenceCache` keyed on `(packageId, version, resolvedTfm)` that holds `AssemblyMetadata` instances and disposes entries above a configurable count cap (default: 200 assemblies). Same snapshot hash → reuse cached references, no re-read. The cache lifetime aligns with `SnapshotStore`.

Do not create a new reference per `ingest_solution` call. The cost of 150+ DLL reads per call will be visible in profiling within the first week of use.

**Warning signs:**
- MCP server process RSS grows across repeated `ingest_solution` calls with no plateau
- Memory growth proportional to package count
- Process uses 1GB+ after 5–10 ingestion calls on a moderately sized solution

**Phase to address:** Memory management design — establish the cache design before wiring reflection into ingestion. This is an architectural decision that is hard to add retrospectively.

### Pitfall 6: Stub Node FQN Does Not Match Reflected Type MetadataName for Generic Types

**What goes wrong:**
Existing stub nodes (created in `SolutionIngestionService.MaybeAddStubNode`) have `FullyQualifiedName` set to `ITypeSymbol.OriginalDefinition.ToDisplayString()`, producing human-readable strings like `System.Collections.Generic.IEnumerable<T>`. The reflected type from `MetadataReference` has `Type.FullName` = `System.Collections.Generic.IEnumerable\`1` (CLR backtick-arity notation). A string equality join produces zero matches for every generic type.

**Why it happens:**
Roslyn display strings are human-readable: `<T>`, `<TKey, TValue>`. The CLR internal format uses `` `1 ``, `` `2 `` suffixes from the ECMA-335 spec. The two formats exist for different audiences and have never been unified. The existing stub synthesis code uses `ToDisplayString()`, which is fine for display but is the wrong key for an enrichment join.

**How to avoid:**
Join on `(ContainingAssemblyName, MetadataName)` tuples:
- From Roslyn stubs: `symbol.ContainingAssembly.Name` + `symbol.MetadataName` (CLR format, e.g. `IEnumerable\`1`)
- From reflected metadata: `type.Assembly.GetName().Name` + `type.Name` (same CLR format)

`ITypeSymbol.MetadataName` is available from Roslyn 4.x (this project uses 4.14.0). It returns the CLR-format name. Build the enrichment index as `Dictionary<(string assembly, string metadataName), SymbolNode>` before iterating reflected types.

**Warning signs:**
- Stub enrichment reports zero matches despite DLLs loading and reflecting successfully
- Enrichment works for `IDisposable` and `Exception` (non-generic) but silently fails for `IEnumerable<T>`, `Task<TResult>`, `IReadOnlyDictionary<TKey, TValue>`
- `find_package_usages` returns no results for commonly-used generic types

**Phase to address:** Stub enrichment — write a unit test matching known stub nodes (created by the existing `SolutionIngestionService` against this solution) against reflected types before building the enrichment pipeline.

### Pitfall 7: Assembly Version Conflicts When Multiple Packages Bring Different Versions of the Same Assembly

**What goes wrong:**
Two packages in the transitive closure each depend on a different version of `System.Text.Json` — say `8.0.5` and `9.0.1`. Loading both DLLs via `MetadataReference.CreateFromFile` into the same Roslyn analysis context causes type identity conflicts: `JsonSerializer` from v8 and `JsonSerializer` from v9 are different objects with the same FQN. Reflection results become ambiguous. In a Roslyn compilation this manifests as CS0433 ("type defined in multiple assemblies").

**Why it happens:**
NuGet's restore already performs version conflict resolution (minimum-version-wins / floating) and writes the single resolved version to `project.assets.json` `targets` section. Code that independently enumerates all packages from the lock file and loads all their DLLs bypasses this resolution.

**How to avoid:**
Derive DLL paths exclusively from `project.assets.json` `targets/{tfm}` resolved package versions — not from the raw lock file's package list. Each package entry in `targets` has the resolved version and a `compile` or `runtime` assets list pointing to the actual DLL name. This is the single source of truth for which assembly version the project actually uses. Load exactly one DLL per unique assembly name per project.

**Warning signs:**
- Two `SymbolNode` entries with the same `FullyQualifiedName` but different `ProjectOrigin` appear after enrichment
- Roslyn compilation using multiple MetadataReferences for the same assembly name emits CS0433
- `find_package_usages` returns inconsistent results depending on traversal order

**Phase to address:** DLL resolution — use `project.assets.json` as the authoritative resolved-version source from the start, not as a fallback.

### Pitfall 8: project.assets.json May Not Exist on Clean Checkout or Before Restore

**What goes wrong:**
`project.assets.json` is used as the dependency source but the `obj/` directory does not exist (clean git clone, Docker multi-stage build before the restore layer, fresh CI agent). The tool fails with a file-not-found exception or returns empty data silently.

**Why it happens:**
`project.assets.json` is a build artifact created by `dotnet restore`. It is correctly excluded from source control in `.gitignore`. It does not exist until restore has been run. In CI pipelines it can be stale if the build uses separate restore and build stages with different working directories.

**How to avoid:**
Before reading `project.assets.json`, check whether it exists. If missing:
- Attempt to run `dotnet restore {projectPath} --no-build` as a subprocess (the MSBuildWorkspace already invokes MSBuild evaluation, so this is acceptable in context).
- If subprocess execution is undesirable, emit a structured diagnostic: `"Dependency graph unavailable: run 'dotnet restore' first"` rather than failing ingestion.

Ingestion of the symbol graph should succeed regardless; only the dependency graph feature is gated on the assets file. `get_dependencies` can return empty with a `missingRestore: true` flag.

**Warning signs:**
- `get_dependencies` returns empty on CI even though the build succeeds
- Works on developer machines (where `obj/` is populated) but fails in fresh Docker containers
- Ingestion succeeds but `get_dependencies` silently returns no data

**Phase to address:** Dependency source detection — write fallback logic and the diagnostic before the first integration test of `get_dependencies`.

### Pitfall 9: Reflecting DLLs Without PathAllowlist Breaks the Existing Security Model

**What goes wrong:**
The new dependency reflection code loads DLLs from the NuGet cache using paths derived from the parsed lock/assets file. If a caller supplies a crafted project path or manipulated assets file that resolves to a DLL outside the NuGet cache or the project tree, `MetadataReference.CreateFromFile` loads an arbitrary binary. The existing `PathAllowlist` pattern secures MCP tool parameters but does not automatically apply to derived DLL paths.

**Why it happens:**
The existing security model (PathAllowlist on tool parameters) was designed for source file paths passed directly by MCP callers. DLL paths are derived internally from parsed file data, creating an indirect code path that bypasses the tool-level checks. This is the same injection vector as SQL injection — trusted input format, untrusted content.

**How to avoid:**
Before calling `MetadataReference.CreateFromFile(dllPath)`, assert that `dllPath` is under the resolved NuGet global cache root (from `SettingsUtility.GetGlobalPackagesFolder()`) or under the project's `obj/` directory. Use the existing `PathAllowlist` infrastructure or a dedicated `NuGetCachePathValidator` that applies the same pattern. Emit an audit log entry when a DLL is loaded, matching the existing AuditFilter pattern.

Filter reflected types to `Type.IsPublic` (and `IsNestedPublic` for nested types) — do not index internal or private types.

**Warning signs:**
- DLL path from a manipulated `project.assets.json` resolves outside the NuGet cache
- Reflected types include internal implementation details from package DLLs
- Audit log has no entries for DLL loads (missing coverage)

**Phase to address:** Security integration — add DLL path validation to the `PathAllowlist` enforcement layer before any reflection code touches a real DLL.

## Technical Debt Patterns

| Shortcut | Immediate Benefit | Long-term Cost | When Acceptable |
|----------|-------------------|----------------|-----------------|
| Hard-code `~/.nuget/packages` as cache root | No NuGet.Configuration dependency | Breaks on CI, Docker, custom `nuget.config` | Never — use `SettingsUtility` or env-var check |
| Load DLLs per `ingest_solution` call without caching | Simple first pass | 250+ MB native heap growth per call; server memory bloat | Never for long-running MCP server |
| Join stub nodes to reflected types on FQN display string | Obvious first attempt | Silent zero-match for all generic types | Never — use `MetadataName` |
| Parse `packages.lock.json` only, ignore `project.assets.json` | Lock file is simpler schema | Fails silently for majority of projects that don't opt in | Only if the tool explicitly documents the opt-in requirement |
| Skip `ref/` folder preference, always use `lib/` | Simpler path construction | Misses public API surface for packages with split ref/impl assemblies | Acceptable for v2.5 with a known-limitation note; fix before v3.0 |
| Load all transitive packages without resolved-version filtering | More complete graph | Assembly version conflicts, identity ambiguity | Never — always use project.assets.json resolved versions |
| No DLL path security check | Faster to implement | Path traversal vector in dependency reflection | Never — add validation before first DLL load |

## Integration Gotchas

| Integration | Common Mistake | Correct Approach |
|-------------|----------------|------------------|
| NuGet cache path | `Path.Combine(userProfile, ".nuget", "packages")` | `SettingsUtility.GetGlobalPackagesFolder(Settings.LoadDefaultSettings(solutionDir))` |
| Lock file TFM key | Match on `"net10.0"` | Match on `.NETCoreApp,Version=v10.0`; use `NuGetFramework.Parse()` then `DotNetFrameworkName` |
| DLL TFM subfolder | Use project TFM directly in path | Best-fit walk: enumerate `ref/` then `lib/` subfolders, filter by compatibility, take highest match |
| Stub-to-reflected-type join | FQN display string equality (`"IEnumerable<T>"`) | `(AssemblyName, MetadataName)` tuple; use `ITypeSymbol.MetadataName` not `ToDisplayString()` |
| Assembly version | Load all lock-file versions | Read resolved versions from `project.assets.json` targets section only |
| Memory management | `MetadataReference.CreateFromFile(path)` per call | `AssemblyMetadata.CreateFromFile(path)` (IDisposable) + cache keyed on `(packageId, version, tfm)` |
| Missing project.assets.json | Throw or return empty silently | Check existence first; emit diagnostic; suggest `dotnet restore` |
| DLL path security | No validation on derived paths | Validate every DLL path is under NuGet cache root via `PathAllowlist` pattern |
| Reflected type scope | Index all types including internal | Filter to `Type.IsPublic` and `Type.IsNestedPublic` |
| packages.lock.json presence | Assume it exists | Treat as optional opt-in; `project.assets.json` is the primary source |

## Performance Traps

| Trap | Symptoms | Prevention | When It Breaks |
|------|----------|------------|----------------|
| Re-loading DLLs on every ingestion call | Ingestion time grows linearly with package count; server memory spikes | `MetadataReferenceCache` keyed on `(packageId, version, tfm)` with LRU eviction cap | First re-ingestion of a solution with 100+ packages |
| Reflecting all transitive packages including dev/test | 50+ test packages (xUnit, Moq, FluentAssertions) reflected unnecessarily | Filter by `PrivateAssets="All"` flag in assets file; skip `developmentDependency` packages | Any project using standard test infrastructure |
| O(n×m) stub enrichment scan without index | Enrichment takes seconds for large solutions | Build `Dictionary<(string, string), SymbolNode>` from stubs before iterating reflected types | Solutions with 20+ projects and many external type references |
| Loading DLLs that are pure reference assemblies (facade) | Extra memory and reflection time with no public types | Filter out facade assemblies (`TypeForwardedTo` only, zero public types) | Packages like `System.Runtime` which are pure type-forward facades |

## Security Mistakes

| Mistake | Risk | Prevention |
|---------|------|------------|
| No validation of DLL path derived from lock file | Path traversal — attacker-controlled assets file forces loading arbitrary DLLs | Validate every DLL path is under `GetGlobalPackagesFolder()` result before `CreateFromFile` |
| Indexing internal/private types from reflected DLLs | Exposes private package internals to MCP query results | Filter to `Type.IsPublic` and `Type.IsNestedPublic` before creating `SymbolNode` entries |
| Logging full DLL paths including username (cache path) | Information leakage in audit log | Redact the cache root prefix when logging; log `packageId/version/lib/tfm/dll.dll` only |
| Accepting DLL path as MCP tool parameter | Arbitrary binary loading | DLL paths are never accepted from callers; always derived internally from ingested project |

## "Looks Done But Isn't" Checklist

- [ ] **Lock file TFM key:** Tested with hand-crafted fixture — verify with a real `packages.lock.json` from `dotnet restore --use-lock-file` on this solution. Confirm the key is `.NETCoreApp,Version=v10.0`, not `net10.0`.
- [ ] **NuGet cache path:** Tested on developer machine — verify on CI with `NUGET_PACKAGES` set to a non-default path (e.g., `D:\a\.nuget\packages` on GitHub Actions Windows).
- [ ] **DLL TFM selection:** Works for `Microsoft.Extensions.Hosting 10.0.0` (ships `net10.0`) — verify for `YamlDotNet 16.3.0` (ships `netstandard2.0` only) where there is no `net10.0` subfolder.
- [ ] **Generic stub enrichment:** Passes for `IDisposable` (non-generic) — verify for `IEnumerable<T>`, `Task<TResult>`, `IReadOnlyDictionary<TKey, TValue>`.
- [ ] **Memory stability:** No leak test run — run 10 consecutive `ingest_solution` calls on the DocAgentFramework solution and confirm process RSS does not grow after the cache warms.
- [ ] **PathAllowlist enforcement:** `get_dependencies` and `find_package_usages` have PathAllowlist on tool parameter — verify the derived DLL paths also pass through a cache-root validation before `CreateFromFile`.
- [ ] **project.assets.json fallback:** Tested with assets file present — test with clean `obj/` directory (delete it) and confirm a clear diagnostic is returned, not an exception.
- [ ] **Version 2 lock files:** Verify the parser handles v2 format (generated by this repo's CPM setup) including the `"version": 2` field and any schema differences.
- [ ] **Assembly version conflicts:** Two packages that share a transitive dependency — verify only the resolved version from `project.assets.json` is loaded, not both.
- [ ] **AssemblyMetadata dispose:** Cache eviction actually calls `.Dispose()` — verify with a memory profiler that native heap does not grow after cache cap is hit.

## Recovery Strategies

| Pitfall | Recovery Cost | Recovery Steps |
|---------|---------------|----------------|
| TFM key mismatch — parser reads empty | LOW | Add `NuGetFramework.Parse()`-based normalizer; update unit tests with real lock file; < 1 hour fix |
| Hard-coded cache path breaks CI | LOW | Replace with `SettingsUtility.GetGlobalPackagesFolder()`; one-line change |
| DLL TFM best-fit missing — FileNotFoundException | MEDIUM | Implement TFM compatibility walk; requires `NuGet.Frameworks` API or local table; 1 day |
| Memory leak from MetadataReference | HIGH | Requires redesigning to `AssemblyMetadata` + cache; affects ingestion pipeline architecture; full retest needed; 2–3 days |
| Stub enrichment zero-match on generics | MEDIUM | Replace join key with `MetadataName` tuples; enrichment logic rewrite + test update; 1 day |
| Assembly version conflict — CS0433 | MEDIUM | Restrict loaded assemblies to `project.assets.json` resolved versions; path resolution code change |
| Missing DLL path security check | MEDIUM | Add `NuGetCachePathValidator` before every `CreateFromFile` call; audit log coverage |
| Missing project.assets.json | LOW | Add existence check + diagnostic emission; no architecture change |
